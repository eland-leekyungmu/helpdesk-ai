# IT Help Desk AI 서비스 상세 요구사항 정의서

> 본 문서는 기존 `help-desk-requirements.md`를 보완하여, 사용자가 제공한 상세 시나리오 및 제약 사항을 구체화한 요구사항 문서입니다.

---

## 1. 프로젝트 개요

### 1.1 목적
회사 IT Help Desk로 접수되는 문의를 AI가 1차로 응답하고, 자동으로 해결되지 않는 문의는 담당 처리자(1차/2차)에게 자동 분배하는 시스템을 구축한다. 기존 1년치 10만 건의 문의-답변 데이터를 RAG로 활용하여 응답 정확도를 확보하고, 피드백 루프를 통해 지속적으로 학습한다.

### 1.2 배경 및 현황
- 회사 임직원 약 **5,000명**이 모든 사업부에서 IT 문의를 접수
- 조직 구조: **법인 → 부서 → 팀** (임직원 및 2차 처리자 모두 이 구조에 소속)
- 접수 창구: **화면 입력** + **이메일**
- **1차 처리자**: 4~5명, 각 사업부의 매뉴얼 기반으로 교육받은 상담 인력
- **2차 처리자**: 각 사업부 소속 회사 직원, 본인 업무가 있는 상태에서 문의를 처리
  - 부서 단위가 아닌 **담당자 단위**로 배정
  - 여러 담당자가 동일 업무에 대해 배정될 수 있음
  - 부서 구성은 변경 가능함
  - 법인 - 부서 - 팀 조직 구조에 소속
- 기존 문의-답변 데이터: **1년치 / 약 10만 건**

### 1.3 핵심 목표
| 항목 | 목표 |
|---|---|
| 1차 응답 해결률 | **70% 이상** |
| 2차 분배 성공률 (올바른 담당자에게 분배) | **90% 이상** |
| 1차 처리 완전 자동화 | 최종 목표 |
| 서비스 가용성 | **24/7 무중단** |
| LLM 호출 비용 절감 | 질문 유형에 따라 모델 차등 호출 |

## 2. 처리 플로우 (핵심 시나리오)

### 2.1 AI 기반 티켓 처리 흐름

```
[사용자 문의 접수]
        │
        ▼
[AI 1차 처리]
  │
  ├─ 기존 1차 처리자 답변과 매칭 가능 → [해당 답변 회신]
  │
  ├─ 2차 처리자에게 넘겼던 건으로 판단 → [적절한 2차 처리자에게 자동 분배]
  │
  └─ 답변을 찾지 못함(신뢰도 < 임계값) → [1차 처리자에게 전달]
```

### 2.2 상세 분기 로직

#### 2.2.1 1차 처리자 답변 회신 (AI 자동)
- AI가 과거 1차 처리자가 답변한 내용을 검색하여 매칭
- 신뢰도(score)가 임계값 이상이면 그대로 회신
- LLM이 현재 문의 맥락에 맞게 답변을 가공하여 전달

#### 2.2.2 2차 처리자 자동 분배 (AI 자동)
- 과거 유사 문의가 2차 처리자에게 전달되어 처리된 이력으로 판단
- 해당 2차 처리자(담당자) 또는 담당자 그룹에 자동 배정
- 배정 근거(유사 과거 티켓 ID, 담당자)를 시스템에 로그

#### 2.2.3 1차 처리자에게 전달 (Fallback)
- AI 응답 신뢰도가 임계값 미만이면 1차 처리자에게 에스컬레이션
- 1차 처리자가 답변하거나 2차 처리자에게 수동 분배

### 2.3 답변 불가 판정 기준
- **신뢰도 점수(score)** 기반 판정 필요 (임계값은 튜닝 가능하도록 설정)
- 후보 기준 (운영 중 조정):
  - RAG 검색 top-k 문서의 유사도 점수
  - LLM 생성 답변의 근거 충분성 평가 점수
  - 답변 불가 판정 시 1차 처리자 Fallback

## 3. 이메일 티켓 처리 규칙

### 3.1 인바운드 이메일 파싱
- 이메일 수신 시 화면 입력과 **동일한 문의 데이터**로 자동 변환
- Zendesk 수준의 **이메일 티켓 식별 기능** 필요
  - 제목/헤더 기반 티켓 ID 매칭
  - 스레드 식별 (Message-ID, In-Reply-To, References)
  - 첨부파일 파싱

### 3.2 이메일 회신 시나리오 처리

| 시나리오 | 처리 방식 |
|---|---|
| 신규 문의 이메일 | AI가 일반 흐름으로 처리 |
| 최종 답변 후 동일 메일에 재문의 | **AI 응답 없이** 이전 답변한 2차 처리자에게 재분배 |
| 2차 처리자가 "본인 업무 아님"으로 회신 | AI가 재분배하거나, 2차 처리자가 직접 다른 담당자로 안내 가능 |
| 한 번 "본인 아님" 처리된 문의 | **재분배 금지** (무한 루프 방지) |

### 3.3 답변 전달 채널
- 요청자가 접수한 경로로 답변 회신
  - 화면으로 접수 → 화면에 답변 표시
  - 이메일로 접수 → 이메일로 답변 회신
- 답변 내용은 **LLM 가공을 통해 생성**

## 4. 메시지 공개 범위 (Public / Private)

### 4.1 기본 원칙
하나의 티켓 내에서 오가는 메시지는 **공개 범위(visibility)** 에 따라 구분되어 저장·조회된다.

| 구분 | 정의 | 참여자 | 요청자 노출 여부 |
|---|---|---|---|
| **Public 메시지** | 요청자와 주고받는 모든 커뮤니케이션 | 요청자 ↔ AI / 1차 처리자 / 2차 처리자 | ✅ 노출 |
| **Private 메시지** | 내부 처리자 간 주고받는 커뮤니케이션 | 1차 처리자 ↔ 2차 처리자, 처리자 ↔ AI 내부 노트 | ❌ 비노출 |

### 4.2 Public 메시지 대상
- 요청자의 최초 문의
- AI가 생성한 자동 답변
- 1차 처리자가 요청자에게 보낸 답변
- 요청자의 추가 질의 및 회신

> **주의**: 2차 처리자는 요청자에게 직접 Public 메시지를 작성할 수 없다. 2차 처리자의 답변은 Private으로 작성되며, AI(LLM)가 자동으로 Public 형태로 가공하여 요청자에게 전달한다.
>
> **AI 가공 규칙**: AI는 2차 처리자의 Private 답변을 Public으로 전환할 때 **문맥(어투, 형식, 문장 구조)만 수정**하며, **내용(사실, 정보, 지시사항, 수치 등)은 절대 변경하지 않는다.**

### 4.3 Private 메시지 대상
- 1차 처리자 → 2차 처리자 이관 코멘트
- **2차 처리자가 작성하는 모든 답변 및 코멘트** (요청자 회신 포함, Private으로만 작성 가능)
- 2차 처리자 간 재이관 코멘트 ("본인 업무 아님" 사유, 다른 담당자 안내 등)
- AI 라우팅 판단 근거, 신뢰도 점수 등 내부 로그성 노트
- 처리자 간 협업을 위한 내부 메모

### 4.4 조회 권한
- **요청자(임직원)**: 사이트 접속 시 본인의 문의에 대한 **Public 메시지만** 조회 가능
  - 티켓 상태, 최종 답변, 요청자-처리자 간 공개 대화 이력 열람
  - Private 메시지와 내부 처리 히스토리는 **비노출**
- **1차/2차 처리자**: 담당 티켓의 Public + Private 메시지 **모두 조회 가능**
- **관리자**: 전체 티켓의 Public + Private 메시지 조회 가능

### 4.5 저장 및 식별
- 모든 메시지는 `visibility` 필드 (`public` / `private`)를 가진다
- API 응답 시 요청자 역할에는 `visibility = public` 메시지만 필터링하여 반환
- 이메일 발송 시 Private 메시지는 요청자에게 절대 발송되지 않도록 **서버 측에서 차단**

### 4.6 UI 요구사항
- 처리자용 화면: Public / Private 메시지를 **시각적으로 구분**하여 표시 (예: 배경색, 아이콘, 라벨)
- 처리자 답변 입력 시 **Public / Private 토글**을 명시적으로 선택하도록 구성
- 요청자용 히스토리 화면: Public 메시지만 시간순으로 노출

## 5. AI / LLM 아키텍처

### 5.1 LLM 호출 전략 (비용 최적화)
| 입력 유형 | 호출 모델 |
|---|---|
| 텍스트 전용 질문 | **경량 LLM** (저비용) |
| 파일/이미지 포함 질문 | **고성능 LLM** (멀티모달) |

- 입력 유형 자동 분류 후 적절한 모델로 라우팅
- **LLM 호출 비용 통계** 제공 (관리자 대시보드)
  - 모델별 호출 건수
  - 모델별 토큰 사용량
  - 기간별 비용 집계

### 5.2 RAG 구성
- **데이터 소스**: 1년치 문의-답변 데이터 (약 10만 건)
- **기술 스택**: AWS Bedrock + Knowledge Base (KB)
- 벡터 검색 기반 유사 문의 매칭 및 답변 생성
- 답변 근거(과거 티켓 ID, 출처) 표시

### 5.3 피드백 루프 (지속 학습)
- 답변 완료된 문의는 학습 데이터로 누적
- 1차 처리자/2차 처리자의 수정 답변도 학습 데이터로 반영
- 오답/수정 사례 레이블링
- KB 재색인 및 재학습 트리거

### 5.4 합성 데이터 생성 (RAG 학습용)
- 기존 실 데이터(1년치 10만 건)를 기반으로 합성/신규 생성 데이터 제작
- 실 데이터를 그대로 복제하지 않고, 동일 유형·패턴의 질문-답변을 새로 생성
- LLM을 활용하여 다양한 변형 생성:
  - Paraphrase (동일 의미, 다른 표현)
  - 시나리오 확장 (기존 케이스의 변형)
  - 상상 기반 신규 케이스 (실 데이터에 없는 유형)
- 생성된 데이터는 RAG Knowledge Base에 적재
- 데이터 품질 검증: 중복 제거, 형식 검증, 카테고리 분류
- 프로젝트 범위에 포함 (MVP 필수)

## 6. 기능 요구사항

### 6.1 사용자(임직원) 기능
- 자연어로 문의 입력 (화면)
- 이메일로 문의 접수 (자동 인입)
- 파일/이미지 첨부 지원
- 답변 수신 (화면 또는 이메일)
- **문의 결과 및 히스토리 조회** (MVP 사이트)
  - 본인 티켓의 **Public 메시지만** 노출 (내부 처리 이력 비노출)

### 6.2 1차 처리자 기능
- AI가 답변 못한 문의 큐 확인
- 직접 답변 또는 2차 처리자에게 분배
- 답변 템플릿/매뉴얼 조회
- **Public / Private 메시지 구분 작성** (요청자 회신 / 내부 이관 코멘트)

### 6.3 2차 처리자 기능
- 본인에게 분배된 문의 수신 (이메일/화면)
- **Private 메시지로만 답변/코멘트 작성 가능** (요청자에게 직접 회신 불가)
- "본인 업무 아님" 회신 처리 (Private)
- 다른 2차 처리자로 안내(수동 재배정, Private 코멘트)
- 2차 처리자의 답변은 AI(LLM)가 문맥만 수정(내용 변경 금지)하여 Public 형태로 요청자에게 자동 전달됨

### 6.4 관리자 기능 (대시보드)
- 전체 티켓 현황 모니터링 (Public + Private 전체 조회 가능)
- **통계**:
  - 1차 자동 응답률
  - 2차 분배 성공률
  - 처리 시간 분포
  - 부서/담당자별 처리 건수
  - **LLM 호출 비용 통계** (모델별/기간별)
- 담당자/부서 관리 (CRUD)
- RAG KB 관리 및 재색인
- 피드백 데이터 관리

## 8. MVP 범위

### 8.1 이번 MVP에 포함
- 간단한 **문의 입력 프론트 화면**
- 백엔드 로직: 문의 접수 → AI 처리 → 답변 또는 팀 분배 결과 표시
- 이메일 인바운드 파싱 및 티켓 식별
- RAG 기반 답변 생성 (Bedrock + KB)
- 1차/2차 분배 로직
- 문의 결과/히스토리 조회 화면 (요청자는 Public 메시지만 노출)
- 처리자용 화면에서 Public / Private 메시지 구분
- 관리자용 통계/대시보드 (기본)
- 인프라 구성 (AWS 기반)

### 8.2 데모로 보여줄 내용
- 문의 입력 → AI 답변 회신 흐름
- 문의 입력 → 2차 처리자 분배 흐름
- 문의 입력 → 1차 처리자 에스컬레이션 흐름
- 요청자 화면에서 Public 메시지만 노출되는 이력 조회
- 관리자 대시보드의 통계 및 LLM 비용 확인

## 9. 개발 구성

### 9.1 팀 구성
- 개발자 **5명**
- **유닛(Unit) 5개**로 분할 개발
- 인프라 구성 포함

### 9.2 유닛 분할 제안 (초안)
> ※ 상세 유닛 분할은 Application Design 단계에서 확정

| # | Unit | 주요 역할 |
|---|---|---|
| 1 | **Frontend Unit** | 임직원 문의 입력 화면, 히스토리 조회, 관리자 대시보드 UI |
| 2 | **Intake & Routing Unit** | 문의 접수 API, 이메일 파싱, 티켓 식별, 분배 엔진 |
| 3 | **AI/RAG Unit** | Bedrock + KB 연동, LLM 라우팅(경량/고성능), 답변 생성 |
| 4 | **Admin & Analytics Unit** | 관리자 인증, 통계/대시보드, LLM 비용 집계, 부서/담당자 관리 |

## 11. 용어 정의

- **1차 처리자**: 각 사업부의 매뉴얼 기반으로 교육받은 상담 인력 (현재 4~5명)
- **2차 처리자**: 각 사업부 소속 담당자. 부서가 아닌 개인 단위로 배정
- **Public 메시지**: 요청자와 주고받는 공개 커뮤니케이션. 요청자 히스토리 화면 및 이메일로 노출
- **Private 메시지**: 내부 처리자 간 주고받는 비공개 커뮤니케이션. 요청자에게 노출되지 않음
- **RAG**: Retrieval-Augmented Generation. 사내 문의-답변 데이터를 검색하여 LLM 답변 생성에 활용
- **KB**: AWS Bedrock Knowledge Base
- **신뢰도 점수(score)**: AI가 답변 가능 여부를 판정하는 기준. 임계값 미만이면 1차 처리자에게 에스컬레이션
- **피드백 루프**: 처리된 문의를 학습 데이터로 축적하여 모델 정확도를 점진적으로 향상시키는 구조
- **티켓 식별**: 이메일 스레드/헤더를 기반으로 동일 티켓을 식별하는 기능 (Zendesk 방식 참고)
