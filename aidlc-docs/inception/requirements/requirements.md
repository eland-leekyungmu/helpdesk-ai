## 2. 기능 요구사항 (Functional Requirements)

### FR-01: 문의 접수
- 임직원이 웹 화면에서 자연어로 문의 입력
- 파일/이미지 첨부 지원
- 이메일로 문의 접수 시 자동 인입 (Amazon SES 인바운드)
- 이메일 티켓 식별: 제목/헤더 기반 티켓 ID 매칭, 스레드 식별 (Message-ID, In-Reply-To, References)

### FR-02: AI 1차 자동 응답
- RAG (Bedrock + KB) 기반 유사 문의-답변 검색
- 텍스트 전용 → 경량 LLM, 파일/이미지 포함 → 고성능 LLM (멀티모달)
- 신뢰도 점수 기반 답변 가능 여부 판정 (임계값: 중간 수준, 튜닝 가능)
- 답변 근거(출처 문서, 과거 티켓 ID) 표시
- LLM이 문맥에 맞게 답변 가공하여 전달

### FR-03: 티켓 자동 분배
- AI가 2차 처리자에게 넘겼던 유사 건으로 판단 시 → 해당 2차 처리자(담당자)에게 자동 배정
- 담당자 단위 배정 (부서가 아닌 개인)
- 여러 담당자가 동일 업무에 배정 가능
- 배정 근거 로그 저장

### FR-04: 1차 처리자 에스컬레이션 (Fallback)
- 신뢰도 임계값 미만 시 1차 처리자에게 전달
- 1차 처리자가 직접 답변 또는 2차 처리자에게 수동 분배

### FR-05: 메시지 공개 범위 (Public / Private)
- 모든 메시지에 `visibility` 필드 (`public` / `private`)
- **Public**: 요청자 문의, AI 자동 답변, 1차 처리자 → 요청자 답변
- **Private**: 1차↔2차 이관 코멘트, 2차 처리자의 모든 작성물, AI 내부 노트
- **2차 처리자는 Private으로만 작성 가능**
- AI가 2차 처리자의 Private 답변을 Public으로 자동 전환
  - **AI 가공 규칙: 문맥(어투, 형식)만 수정, 내용(사실, 정보, 지시사항) 변경 금지**
- 요청자는 Public 메시지만 조회 가능
- 이메일 발송 시 Private 메시지 요청자 노출 서버 측 차단

### FR-06: 이메일 회신 시나리오
- 최종 답변 후 동일 메일에 재문의 → AI 응답 없이 이전 2차 처리자에게 재분배
- 2차 처리자 "본인 업무 아님" 회신 → AI 재분배 또는 수동 안내
- 한 번 "본인 아님" 처리된 문의 → 재분배 금지 (무한 루프 방지)

### FR-07: 사용자 히스토리 조회
- 임직원이 사이트 접속 시 본인 문의의 Public 메시지만 조회
- 티켓 상태 (접수/진행중/완료) 확인
- Private 메시지 및 내부 처리 이력 비노출

### FR-08: 처리자 화면
- 1차 처리자: 미처리 문의 큐, Public/Private 구분 작성, 매뉴얼 조회
- 2차 처리자: 분배된 문의 수신, Private 전용 작성, "본인 아님" 처리, 수동 재배정
- Public/Private 시각적 구분 (배경색, 라벨)
- 처리자 답변 입력 시 Public/Private 토글 (1차만 Public 가능)

### FR-09: 관리자 대시보드
- 전체 티켓 현황 모니터링 (Public + Private 전체 조회)
- 통계: 1차 자동 응답률, 2차 분배 성공률, 처리 시간, 부서/담당자별 건수
- LLM 호출 비용 통계 (모델별 호출 건수, 토큰 사용량, 기간별 비용)
- 담당자/부서 관리 (CRUD)
- RAG KB 관리 및 재색인

### FR-10: 피드백 루프
- 답변 완료 문의 → 학습 데이터로 자동 누적
- 처리자 수정 답변도 학습 데이터 반영
- 오답/수정 사례 레이블링
- KB 재색인 및 재학습 트리거
- 답변 유용성 피드백 (👍/👎)

### FR-11: 인증 및 권한
- 자체 DB 기반 사용자 관리 (이메일 + 비밀번호)
- 비밀번호 안전 저장 (bcrypt 해싱)
- Role 기반 권한 분리: 임직원 / 1차 처리자 / 2차 처리자 / 관리자
- 동일 인증 시스템에서 역할로 구분
- 로그인 시도 횟수 제한

### FR-11.1: 조직 구조 관리
- 모든 사용자(요청자, 2차 처리자 포함)는 **법인 → 부서 → 팀** 계층 구조에 소속
- 조직 구조는 관리자가 CRUD로 관리 (법인/부서/팀 추가·수정·삭제)
- 사용자는 반드시 하나의 팀에 소속 (팀 → 부서 → 법인으로 상위 조직 자동 결정)
- 2차 처리자 분배 시 조직 정보를 참고하여 적절한 담당자 매칭 가능
- 통계/대시보드에서 법인별, 부서별, 팀별 문의 건수 집계 가능

### FR-12: LLM 비용 최적화
- 텍스트 전용 → 경량 LLM 호출
- 파일/이미지 포함 → 고성능 멀티모달 LLM 호출
- 입력 유형 자동 분류 후 모델 라우팅
- 비용 통계 대시보드 제공

### FR-13: 합성 데이터 생성 (RAG 학습용)
- 기존 실 데이터(1년치 10만 건)를 기반으로 합성/신규 생성 데이터 제작
- 실 데이터를 그대로 복제하지 않고, 동일 유형·패턴의 질문-답변을 새로 생성
- LLM을 활용하여 다양한 변형(paraphrase, 시나리오 확장, 상상 기반 신규 케이스) 생성
- 생성된 데이터는 RAG Knowledge Base에 적재하여 학습 데이터로 활용
- 데이터 품질 검증 프로세스 포함 (중복 제거, 형식 검증, 카테고리 분류)

## 5. MVP 범위

### 포함
- 문의 입력 프론트 화면 (React/Next.js)
- 백엔드 API (NestJS)
- AI 자동 응답 + 분배 로직
- 이메일 인바운드 파싱 및 자체 티켓 식별
- Public/Private 메시지 구분
- 사용자 히스토리 조회 (Public만)
- 처리자 화면 (1차/2차)
- 관리자 대시보드 + LLM 비용 통계
- 피드백 루프 (전체)
- **합성 데이터 생성 (실 데이터 기반 10만 건 규모 RAG 학습 데이터)**
- 인프라 (Terraform, AWS)
- 자체 인증 시스템 (Role 기반)

### 제외
- Zendesk 외부 연동
- 실시간 알림 (MVP에서는 수동 새로고침)
- 다국어 지원

## 7. 제약 사항 및 특이사항

- 2차 처리자는 Public 메시지 작성 불가 (Private 전용)
- AI 가공 시 내용 변경 금지 (문맥/어투만 수정)
- "본인 아님" 1회 처리 후 재분배 금지
- 이메일 최종 답변 후 재문의 시 AI 미개입, 이전 2차 처리자에게 직접 재분배
- 신뢰도 임계값은 운영 중 튜닝 가능하도록 설정값으로 관리
